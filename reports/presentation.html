3. 모듈 ①: 데이터 불균형 고도화 전략

왜 SMOTE를 쓰지 않았는가?

SMOTE는 연속형 수치 공간의 선형 보간 기법입니다. 하지만 고차원 텍스트 임베딩 공간에서 두 벡터의 중간점은 의미 없는 노이즈가 될 우려가 큽니다.

솔루션: LLM 기반 문맥 맞춤형 합성 (Synthesis)

  • GPT-4o를 활용해 '그루밍', '자해 신호' 등 15,000건의 유효한 문맥을 지닌 긴급 데이터 합성.
  • 이는 수업 Case Study 3에서 cWGAN(조건부 적대 생성 신경망)을 통해 내재적 데이터 분포를 학습하여 오버샘플링한 것과 완벽히 동일한 방법론적 발전입니다.
  • (품질 검증: 템플릿 기반 구문 통제 및 Hold-out 평가 FPR 0.0 달성)

5. 해석 가능한 AI (SHAP) 도입 및 통찰

[Local SHAP] 모듈 ① 텍스트 분류

단순 막대 그래프를 넘어, "전체 긴급 예측 기여도 Top-5 키워드" 분석 후 오분류 1건을 심층 해석(입체성 확보).

정책 제언: '비밀', '내가 다 해줄게' 등의 그루밍 토큰 기여도가 높음. 복지사 대시보드에 해당 키워드 룰을 삽입해 우선 모니터링 대상으로 알림을 띄우는 설계가 필요.
[Global SHAP] 모듈 ② 매칭 랭킹
정책 제언: f_cosine(소개글 유사도)이 높을수록(빨간점) 매칭 호환성을 (+) 강하게 견인. UX적으로 유저의 상세 소개글 작성을 유도해야 함. 장애 일치 여부(f_dis_match)는 절대적이지 않아 교차 추천 가능.

6. 모듈 ②: Data Leakage 및 과적합 제어

1. Data Leakage (데이터 누수) 원천 차단

  • User-level Disjoint Split: Train과 Test 세트에 동일한 사용자가 등장하지 않도록 분리.
  • Label Leakage 차단: 매칭 정답에 직결되는 '지역 일치' 등의 변수를 의도적으로 제거(mode=embedding).
  • Frozen SBERT: 임베딩 추출기 Fine-tuning 제외.

2. 고차원 임베딩의 Feature Selection

Q. SBERT 고차원 벡터를 그대로 쓰면 과적합되지 않나?
본 프로젝트는 트리의 앙상블인 LightGBM을 사용했습니다. LightGBM은 자체적인 정보 획득량(Information Gain) 기반 분기를 수행하므로 모델 내부적으로 Feature Selection이 이루어집니다. 이에 더해 L1/L2 정규화(Regularization) 파라미터를 강하게 주어 과적합을 제어했습니다.

Appendix: 예상 Q&A Backup

Q1. 특성 선택(Feature Selection)을 별도로 하지 않은 이유?

👉 LightGBM 트리의 내장된 분기 중요도(Split gain) 연산으로 암묵적 특성 선택이 이루어집니다. 고차원 텍스트 벡터 전체의 비선형 조합이 매칭에 필요하기에, 강제 삭제(SelectFromModel 등)보다는 L1/L2 정규화로 과적합을 제어하는 것이 효과적이라고 판단했습니다.

Q2. 모듈 1과 모듈 2의 시스템적 통합 방안은?

👉 독립된 모듈이 아닙니다. 모듈 1에서 경고/긴급 판정을 받은 유저는 '위험도 스코어'가 누적되며, 이 스코어는 모듈 2 랭킹 모델의 '음의 가중치(Penalty)'로 작용하여 건강한 유저들의 추천 리스트에서 즉각적으로 배제되는 안전 루프를 형성합니다.

Q3. 데이터 극단적 불균형 전, Naive Baseline의 정확도는?

👉 긴급 0%인 42:20:38:0 분포에서 ML 없이 무조건 '정상(42%)'으로만 예측할 경우 기준 정확도는 41.9%에 불과합니다. 본 프로젝트는 이 Naive Baseline을 돌파하여 의미 있는 Macro-F1 0.76을 확보한 것입니다.
